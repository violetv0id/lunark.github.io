Networking Basics

Learn common Networking, understand RPCs, and figure out which provider works best for you.

Getting Started

Multiplayer in video games can easily make or break the experience - That's why it's important to plan ahead and choose your path before spending months on it.

This Article goes into how some common Networking things work internally, alongside what Multiplayer Provider might work best for your project.

Picking a Provider

Let's start simple. Physics-oriented games typically need a 'Shared' topology.
This means that the host does not own every object, but rather every player owns their own objects.
If someone picks a prop up, it becomes theirs, and everyone else replicates it.
This is usually pretty unsafe, so it requires a lot more guardrails.

For competitive games, you'd typically want to trust the player as least as possible, as if the host is a cheater themselves, not only do they control the server, but they control their own player, too.
Competitive games in general would need a Dedicated-Serverhost architecture. You host servers so that nobody plays both Server and Player.

For casual games, it really depends on what you want.
Free? Fishnet, Mirror, maybe Purrnet if you're using Unity.
Willing to pay post-release? Photon is likely the best way to go.

Understanding RPCs

In order to understand RPCs, you first have to understand what they are.
RPCs - short for Remote Procedural Calls - are methods that get called remotely.

For example, the host could call an RPC called 'RPC_StartRound', and every player that receives the RPC runs whatever code is within the RPC's method.
As the name suggests, it's really just a way to tell everyone to call the same method.

Here's an example: 


int i = 0;
[RPC(RpcSources.StateAuthority, RpcTargets.All)]
public void RPC_LogIndex()
{
    i++;
    Debug.Log(i);
}

In this example, every player would locally run this method, and log however many times it has been called on their client in the past, including this time.
